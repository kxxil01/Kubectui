# Specification: Real Performance Bottlenecks

**Track ID:** perf-bottlenecks_20260312
**Type:** Bug
**Created:** 2026-03-12
**Status:** Draft

## Summary

Address the two remaining real performance bottlenecks: (1) full deep clone of GlobalState/ClusterSnapshot on every refresh cycle, and (2) semaphore contention between wave 1 (core) and wave 2 (secondary) resources during refresh.

## Context

The M15b performance analysis identified 4 bottlenecks. Two are already shipped:
- **Parallel detail API calls** — already uses `tokio::join!` (`main.rs:4709`)
- **Background context connect** — already uses `tokio::spawn` (`main.rs:2398`)

Two remain:
- **Snapshot deep clone** — `global_state.clone()` at `main.rs:887` copies 30+ Vec<T> fields (500KB–2MB on large clusters) every refresh (default 30s)
- **Refresh wave contention** — Wave 1 (14 core resources) and Wave 2 (22 secondary resources) share a single 8-permit semaphore; secondary fetches can starve core resources

## Problem Description

### Snapshot Deep Clone
- `spawn_refresh_task()` takes `GlobalState` by value (line 825)
- Call site clones: `global_state.clone()` (line 887)
- `ClusterSnapshot` has 30+ `Vec<T>` fields, each Vec clones its entire backing buffer
- This allocation pressure occurs every 30s auto-refresh, plus manual F5, plus namespace/view switches
- The clone exists because the refresh task mutates the snapshot in-place, then returns it via channel

### Refresh Wave Contention
- `RESOURCE_FETCH_SEMAPHORE` has 8 permits (`MAX_CONCURRENT_RESOURCE_FETCHES`, state/mod.rs:28)
- When `include_secondary_resources=true`, 36 total fetches compete for 8 permits
- Wave 2 fetches can acquire permits before Wave 1 fetches, delaying core resources
- Both waves run concurrently via `tokio::join!(wave1, wave2)` (line 1476), but the semaphore doesn't prioritize wave 1
